Observatory opportunities

Optional capabilities assessed independently from upgrade necessity.

prototype

Extend metadata-only project enrichment beyond Claude and Codex

medium effort

Unified ccusage reports already ingest 15 supported agent sources, but Observatory project paths, stable session enrichment, and project activity currently cover only Claude and Codex. Adding selected high-value sources would turn existing aggregate rows into project-level analytics.

Architecture fit

Fits the local-only architecture if each added source gets an explicit metadata-head parser, stable native-key mapping, and regression fixtures without copying prompt or response content.

Work outline

  • Choose sources based on actual local usage rather than implementing all agents at once.
  • Add source-specific path discovery and native session-key extraction in server/path-indexer.ts.
  • Expand provider/project types and identity fixtures while preserving annotation continuity.
  • Surface unsupported enrichment explicitly instead of presenting missing paths as collection failure.
Supported upstream sources↗
adopt

Expose an explicit reporting timezone

low effort

ccusage supports IANA timezones, while the Observatory currently accepts the process/system timezone implicitly. An explicit setting would make day, week, and month boundaries deterministic when the app moves between machines or environments.

Architecture fit

This remains local and can be passed as a CLI argument alongside --offline. The same setting must also drive Observatory-side local-date grouping to avoid mixed boundaries.

Work outline

  • Add and validate an IANA timezone setting.
  • Pass --timezone to unified and block commands.
  • Use the same timezone in project aggregation and document the boundary.

prototype

Add governed local pricing overrides for newly released models

medium effort

Upstream configuration supports runtime pricing overrides keyed by raw model name. A controlled override could prevent temporary zero or stale API-equivalent costs while bundled offline pricing catches up.

Architecture fit

Compatible with offline operation, but the Observatory must label overrides separately, record their source and effective date, and avoid implying that user-supplied values are upstream ccusage defaults.

Work outline

  • Use an Observatory-owned explicit config path instead of relying on ambient auto-discovery.
  • Validate model names and numeric pricing fields against the bundled schema.
  • Expose override provenance in methodology and source health.
  • Add cost regression fixtures before enabling an override.
